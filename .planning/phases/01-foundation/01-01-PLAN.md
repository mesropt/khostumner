## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer machine -> Docker daemon | docker-compose.yml and .env read from developer's filesystem; compromised .env exposes DB credentials |
| docker network -> postgres service | Backend connects to postgres using hardcoded dev credentials; not exposed outside Docker network in dev |
| public internet -> backend | GET /health is the only public endpoint in this plan; no authentication required |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | docker-compose.yml + .env | mitigate | Use env_file: [.env] for postgres credentials in docker-compose.yml; add .env to .gitignore; provide .env.example with placeholder values |
| T-01-02 | Information Disclosure | DB_ECHO setting | mitigate | DB_ECHO defaults to false; controlled via env var in config.py; never set true in .env.example |
| T-01-03 | Spoofing | CORS middleware | mitigate | allow_origins restricted to ["http://localhost:5173"] in dev — not wildcard "*" |
| T-01-04 | Denial of Service | Connection pool exhaustion | accept | pool_size=10, max_overflow=20 is adequate for dev; production tuning deferred to deployment phase |
| T-01-05 | Tampering | Alembic migration files | accept | Migration files are version-controlled; autogenerate from trusted models; no user input involved |
| T-01-SC | Tampering | pip install from requirements.txt | mitigate | All packages verified via PyPI pip index versions in RESEARCH.md Package Legitimacy Audit; no [ASSUMED] or [SUS] packages in backend requirements |
